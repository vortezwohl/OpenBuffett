## Context

当前 runtime 已经解决了 Strands 对 schema-driven 工具的错误包装问题，但 fileglide 在“模型可用性”上仍然存在系统性短板：

- 工具回给模型的结果过度依赖 `summary`，目录列举、文件列举和文本读取在模型侧常被压缩成工具名或路径名，导致后续推理缺乏真实上下文。
- fileglide wrapper 直接暴露 `root`、`start`、`target`，但没有把它们的语义差异转成模型容易遵守的契约，绝对路径误填后只能落回底层 `ScopeError`。
- 遍历类工具的默认递归策略对 Windows 根盘不友好，模型首次探索磁盘时容易直接撞进 `$RECYCLE.BIN`、`System Volume Information` 等权限边界。
- 错误虽然能进 timeline，但诊断文本还停留在“人类开发者能看懂”的层级，没有为模型提供下一步修正动作。

约束：

- 不修改 fileglide facade 与其底层服务行为；
- 不引入新依赖，不重建 agent 架构；
- 保持现有 `ToolSpec` / `ToolContext` / `ToolResult` 主体结构；
- 优先修只读探索闭环，写入类工具后续按同一协议扩展。

相关角色：

- runtime 负责把工具协议准确回给模型；
- fileglide wrapper 负责输入规范化、默认值和可恢复错误；
- TUI/workbench 负责维持“先列举、再缩小范围、再读取”的稳定默认路径。

## Goals / Non-Goals

**Goals:**

- 让模型拿到可操作的工具正文，而不是只有摘要占位文本。
- 让只读 fileglide 工具形成稳定、统一、可学习的路径契约。
- 让常见绝对路径误用在 wrapper 层被自动规范化或被明确拒绝。
- 让默认探索行为更安全，避免第一次列举根盘就触发权限雪崩。
- 让失败文本兼顾 timeline 可见性与模型纠错性。

**Non-Goals:**

- 不在这一轮扩展全部 fileglide 写入/批处理工具的契约。
- 不重写 fileglide 的异常体系或 traversal 实现。
- 不在这一轮引入复杂权限模型、策略引擎或 per-tool prompt 模板系统。
- 不以“完全隐藏底层错误”为目标，底层错误仍需保留给诊断和测试。

## Decisions

### 1. 分离“模型正文”和“UI 摘要”两条结果通道

runtime 不再把 `ToolResult.summary` 视为模型侧的默认结果正文。模型侧应收到结构化内容序列化后的稳定正文；UI/timeline 仍然可以继续使用摘要、预览和折叠详情。

原因：

- `summary` 的职责是概览，不是完整语义；继续复用会让模型只看到 `path.list`、`file.list` 或文件路径。
- 模型要做下一步工具决策，必须能直接消费目录项、文件内容和错误建议。

备选方案：

- 继续沿用 `summary`，只在个别工具里补更多文本。放弃，原因是会让正确性分散到每个工具实现里，长期不可维护。

### 2. 在 wrapper 层建立只读工具的规范化输入契约

对 `path.list`、`file.list`、`tree.list`、`text.read`、`text.grep` 等只读工具，引入一层轻量规范化：

- 若 `root` 缺失且 `start`/`target` 为绝对路径，则自动拆分为“锚点根目录 + 相对路径”。
- 若 `root` 已提供且绝对路径仍落在该 root 内，则允许归一化为相对路径后继续执行。
- 若 `root` 已提供但绝对路径逃逸该 root，则直接返回显式契约错误，而不是把通用 `ScopeError` 原样抛给模型。

推荐锚点规则：

- Windows 使用盘符根，如 `D:\`
- POSIX 使用 `/`

原因：

- 这是最小、最稳定的跨平台锚点；相对路径始终相对于明确根，后续结果里的 `relative_path` 也更一致。
- 对模型而言，“绝对路径被拆成 root+relative path”比“你输错了自己猜哪里错”更可恢复。

备选方案：

- 完全不自动归一化，只靠报错和提示词训练模型。放弃，原因是这会把同类错误反复留给模型试错。
- 把 `root` 直接设为目标文件父目录。放弃，原因是会让目录列举与文件读取的相对路径基准不稳定。

### 3. 把安全探索策略固化为默认行为，而不是靠 prompt 暗示

默认 workbench 里的列举型工具应遵循更保守的探索路径：

- `path.list` / `file.list` 默认 `recursive=false`
- 根盘或锚点根目录的首次探索默认只看当前层
- 模型若要深入，再显式增加 `start` 或开启 `recursive=true`

原因：

- Windows 根盘权限噪音不是个别异常，而是默认环境事实。
- “先看一层，再缩小，再深入”更符合真实排障和阅读流，也更节省 token。

备选方案：

- 只对 `D:\` / `C:\` 特判。放弃，原因是规则过窄，不利于跨平台和后续 WebUI 复用。
- 保持递归默认值不变，只在失败后提示重试。放弃，原因是第一次失败本身就会扰乱模型状态。

### 4. 失败反馈分为原始诊断与纠错建议两层

wrapper 应保留底层原始错误用于 timeline metadata 和测试断言，同时生成更面向模型的稳定失败文本。失败文本至少区分：

- 契约误用：绝对路径放错槽位、`root` 与目标冲突
- 作用域违规：目标逃逸当前 root
- 目标不存在：起始路径或文件不存在
- 权限受限：遍历进入系统目录或无权读取

每类失败都应附最小纠错建议，例如：

- “若要读取 `D:\foo\bar.txt`，请传 `root=\"D:\\\"`，`target=\"foo/bar.txt\"`”
- “当前是根盘探索，请先使用 `recursive=false` 缩小范围”

原因：

- timeline 关注发生了什么，模型关注下一步怎么修；
- 这能把工具失败从“终点”变成“可恢复分支”。

备选方案：

- 只返回中文包装错误，不保留原始异常。放弃，原因是会削弱诊断能力。

### 5. 用协议层测试锁住模型真实可恢复路径

新增测试不只断言“工具能跑”，还要覆盖：

- 目录列举正文会回给模型
- 文本读取正文会回给模型
- 绝对路径会被正确规范化
- 冲突路径会得到明确契约错误
- 根盘列举默认不会直接递归
- 权限失败文本能指导下一步重试

原因：

- 这类问题核心在“模型有没有足够信息做下一步”，只测 handler 成功不够。

## Risks / Trade-offs

- [Risk] 模型侧正文变长，单次工具结果 token 成本上升  
  → Mitigation: 只对模型需要决策的正文保真；UI 仍使用预览与折叠详情，必要时对超大结果做稳定裁剪。

- [Risk] 自动归一化可能掩盖调用方对 root 语义的误解  
  → Mitigation: 仅对无歧义只读场景归一化；一旦 root 与目标冲突就显式失败。

- [Risk] 默认非递归可能让部分“一步到位搜索全盘”的用例变慢  
  → Mitigation: 只调整列举型工具默认值；显式 `recursive=true` 仍可开启深度探索。

- [Risk] 包装后的纠错文本可能与底层 fileglide 错误语义漂移  
  → Mitigation: 在 metadata 中保留原始异常和错误类别，测试同时覆盖用户态文本与底层诊断。

## Migration Plan

1. 先调整结果序列化路径，拆分模型正文与 UI 摘要。
2. 在 fileglide wrapper 中引入只读工具的路径规范化与契约级错误包装。
3. 调整列举型工具默认值，并同步更新 workbench 系统提示与默认探索描述。
4. 为 timeline 保留原始错误元数据，同时让模型收到更稳定的失败文本。
5. 增补 Windows 根盘、绝对路径和结果正文相关测试。

回滚策略：

- 若模型正文回流引发意外副作用，可先只回滚结果保真改动，保留路径契约与安全默认值；
- 若路径规范化引入误判，可切回“显式报契约错误，不做自动归一化”的保守路径。

## Open Questions

- 对 `text.grep` 这类可能返回大结果的工具，模型正文是否需要单独的截断/分页协议，而不是直接复用 list/read 的保真策略。
- 这一轮是否把 `path.search` / `file.search` 也纳入同一规范化层；原则上建议纳入，但实现时可以按只读高频工具优先级排队。
- 错误建议文本是否统一中文；当前项目默认中文更合适，但若后续多语言 UI 出现，可能需要结构化错误码加本地化层。
