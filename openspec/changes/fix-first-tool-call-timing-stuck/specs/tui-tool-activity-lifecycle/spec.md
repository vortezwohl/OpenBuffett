## ADDED Requirements

### Requirement: 单条工具调用必须只对应一条可收口的 timeline 活动项
系统 MUST 让同一条工具调用从 `started` 到终态始终关联到同一条工具活动项，不得在终态到来后同时保留一条仍在计时的旧 `Running` 行和一条新的终态行。

#### Scenario: 首条工具调用在同批次后续工具开始后仍能正确收口
- **WHEN** 第一条工具调用已经进入 `started` 状态，且同批次第二条工具调用在第一条结束前也进入 `started`
- **THEN** 第一条工具调用收到终态事件后，系统 MUST 把该终态回填到原始第一条工具活动项
- **AND** 系统 MUST NOT 继续保留第一条工具调用的持续计时 `Running` 行

#### Scenario: 同名工具的连续调用不会串线
- **WHEN** 同一批次中两条 `name` 相同但 `tool_use_id` 不同的工具调用先后开始并分别结束
- **THEN** 系统 MUST 依据调用实例区分这两条工具活动
- **AND** 每条调用 MUST 只生成一条最终收口的 timeline 行

### Requirement: 工具终态必须停止本地计时
系统 MUST 在工具活动进入 `completed`、`failed` 或 `cancelled` 后清除该活动项的运行态，使本地计时停止并固定为最终耗时。

#### Scenario: 成功工具调用停止计时
- **WHEN** 某条工具调用收到 `completed` 终态
- **THEN** 该工具活动项 MUST 停止继续增长 `duration`
- **AND** 该工具活动项 MUST 显示最终成功状态

#### Scenario: 失败工具调用停止计时
- **WHEN** 某条工具调用收到 `failed` 终态
- **THEN** 该工具活动项 MUST 停止继续增长 `duration`
- **AND** 该工具活动项 MUST 显示失败状态与对应错误摘要

#### Scenario: 取消时所有活跃工具都被收口
- **WHEN** 当前 turn 在存在一个或多个活跃工具调用时被取消
- **THEN** 系统 MUST 为每条活跃工具活动生成已取消终态
- **AND** 系统 MUST NOT 留下继续计时的工具活动项

### Requirement: 终态关联失败时系统必须执行保守回绑而不是直接遗留运行项
当工具终态事件无法通过首选实例键精确命中原始活动项时，系统 MUST 执行保守回绑或等价兜底，以优先收口仍在运行的原始工具活动项。

#### Scenario: 终态缺少首选键时优先回绑运行中的原始活动项
- **WHEN** 工具终态事件无法通过首选实例键直接命中活动项，但存在一条仍在运行且可由辅助信息识别的原始工具活动项
- **THEN** 系统 MUST 优先把终态回填到该原始活动项
- **AND** 系统 MUST NOT 仅因为精确键缺失就额外新建一条终态行并保留旧运行项
