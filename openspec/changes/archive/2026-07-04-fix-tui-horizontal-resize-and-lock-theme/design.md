## Context

当前 `AgentWorkbenchApp` 的主界面由 `Header`、状态栏、`VerticalScroll` timeline、queue tray 和输入区组成，消息内容通过本地 `_refresh_view()` 系列方法写入多个 `Static` 组件。现有实现对“收到新事件后刷新”覆盖较好，但没有把“终端宽度变化后整页需要重排并重绘”当成明确的界面生命周期来处理。

从现有 CSS 和截图现象看，问题更像是表面绘制与宽度收缩之间的契约不完整，而不是业务异常：
- 部分带边框区域依赖透明底板；
- 主界面缺少显式的横向收缩约束；
- 本地渲染内容主要在事件流到来时刷新，宽度变化后缺少一次确定性的全量重绘。

另一方面，TUI 目前直接继承 Textual 默认主题系统。`App.get_system_commands()` 会暴露 `Theme` 命令，`action_change_theme()`、`search_themes()` 和 `action_toggle_dark()` 也都可用。这与“界面必须固定为单一视觉基线”的新要求冲突。

## Goals / Non-Goals

**Goals:**
- 让 TUI 在终端宽度缩窄和恢复时都能稳定重排，不出现旧边框残影、断裂线段或错位内容。
- 明确主界面哪些区域负责绘制深色表面，避免宽度变化后出现未被新表面覆盖的旧像素区域。
- 在应用内部把主题显式固定为 `ansi-dark`，确保首屏和后续重绘都建立在同一调色基线上。
- 关闭用户可触发的主题切换路径，而不是仅仅“默认先用 `ansi-dark`”。
- 为横向 resize 和主题锁定补齐自动化测试。

**Non-Goals:**
- 不引入新的主题配置系统，也不支持多主题切换。
- 不重做现有 timeline 语义、事件协议或聊天视觉语言。
- 不把这次修复扩展成整套复杂断点布局系统。
- 不处理与本次需求无关的终端交互问题。

## Decisions

### 1. 把横向 resize 视为显式的界面刷新事件，而不是依赖被动重绘

决策：为 `AgentWorkbenchApp` 增加宽度变化后的显式刷新路径，在终端尺寸变化时重新刷新状态栏、timeline 和 queue tray，并触发必要的重排/重绘。

原因：
- 当前本地渲染内容主要在事件流更新时重算，窗口变窄本身不会带来新的业务事件；
- 截图中的残影说明仅靠 Textual 默认布局刷新并不足以保证这些 `Static` 内容和边框全部按新宽度重画；
- 将 resize 纳入应用自己的刷新闭环后，问题定位和测试都更直接。

备选方案：
- 完全依赖 Textual 的默认 resize 行为，不增加应用级刷新。
  不采用，因为现象已经表明默认行为没有覆盖当前组合方式下的全部重绘需求。

### 2. 由应用显式拥有主界面的暗色表面，避免透明区域在缩宽后留下旧边框残影

决策：为 `Screen`、主容器和关键带边框区域建立一致的深色表面绘制策略，并补充必要的横向收缩约束，例如让可收缩容器显式以当前可用宽度布局。

原因：
- 现有界面大量使用边框线，但多个大容器没有明确背景色，旧宽度下绘制过的边框有机会在收缩后残留；
- 一旦主题固定为 `ansi-dark`，界面拥有稳定的深色底板，可以安全地由应用负责完整表面绘制；
- 这比继续依赖透明宿主背景更适合做可预测的 resize 验证。

备选方案：
- 继续沿用透明底板，只尝试手动刷新单个问题组件。
  不采用，因为这样无法给整页表面建立统一契约，后续仍容易在别的带边框区域复发。

### 3. 在应用类内部固定 `ansi-dark`，而不是把默认主题留给入口或用户环境

决策：由 `AgentWorkbenchApp` 在初始化阶段显式设置主题为 `ansi-dark`，使首屏渲染、测试环境和真实运行环境都共享同一默认值。

原因：
- 主题是界面行为的一部分，不只是启动参数；
- 如果把默认值留在外部入口或环境层，测试和实际运行更容易偏离；
- 主题固定后，后续颜色和表面策略才能建立在单一已知前提上。

备选方案：
- 仅在 `__main__` 入口设置主题。
  不采用，因为测试直接实例化 `AgentWorkbenchApp` 时仍可能绕过该默认值。

### 4. 同时关闭主题命令和主题切换 action，避免只封一个入口

决策：在应用类中收口所有用户可达的主题切换入口，包括：
- 从 `get_system_commands()` 结果中移除 `Theme`；
- 覆写 `action_change_theme()`，不再打开主题搜索；
- 覆写 `search_themes()`，防止其他路径直接唤起主题面板；
- 覆写 `action_toggle_dark()`，保证即使 action 被触发也不会改变当前主题。

原因：
- Textual 的主题切换入口不只一个，只关掉命令面板项是不够的；
- “不允许用户自己设置”本质上要求运行中的 TUI 不提供任何可达的换肤能力；
- 统一在 `AgentWorkbenchApp` 内处理，边界最清晰。

备选方案：
- 只移除命令面板中的 `Theme` 命令。
  不采用，因为 `action_toggle_dark()` 和直接调用 `search_themes()` 仍会留下旁路。

### 5. 用 `Pilot.resize_terminal(...)` 和 App 行为断言做验证，而不是只看静态字符串

决策：测试分成两类：
- resize 测试：通过 `Pilot.resize_terminal(...)` 驱动宽度缩放，断言关键组件宽度会随之收缩、界面仍可刷新、消息文本在缩宽后继续可见；
- 主题锁定测试：断言应用启动后主题为 `ansi-dark`，系统命令中不再暴露 `Theme`，主题切换 action 触发后主题保持不变。

原因：
- 这类问题如果只检查 `_render_timeline_text()`，无法覆盖真正的终端尺寸变化；
- `Pilot` 已提供标准终端 resize 能力，适合把这次 bug 转成稳定回归测试。

备选方案：
- 仅做手工截图验证。
  不采用，因为无法形成可重复的回归保护。

## Risks / Trade-offs

- [Risk] 为修复 resize 残影而补上的深色表面，可能与此前“更接近宿主终端原生底板”的方向冲突。 → Mitigation：本次以“固定 `ansi-dark` 且稳定可重绘”为更高优先级，只在必要区域持有暗色表面，不额外扩大装饰面。
- [Risk] 仅靠一次应用级刷新仍不足以消除所有残影。 → Mitigation：把“显式刷新”和“显式表面覆盖”一起落地，避免单点修补。
- [Risk] 关闭主题切换 action 后，未来若要恢复多主题能力需要再开接口。 → Mitigation：这属于当前需求明确要求的约束，后续如要放开，应通过新 change 明确重设计。
- [Trade-off] 这次不做复杂断点布局，只保证收缩稳定和内容可读。 → 这是刻意收敛，优先解决“缩宽会坏”和“主题不可控”两个核心问题。

## Migration Plan

1. 先在 `AgentWorkbenchApp` 内固定 `ansi-dark`，并收口主题切换命令与 action。
2. 再调整主界面容器的表面绘制与横向收缩约束，并补上 resize 后的显式刷新路径。
3. 最后增加 `Pilot.resize_terminal(...)` 与主题锁定测试，确认宽度变化和主题不可变都能稳定回归。

回滚策略：
- 若 resize 修复引入新的表面问题，可先回滚容器背景与 resize hook；
- 若主题锁定造成预期外行为，可单独回滚主题相关覆写，而不影响 resize 修复。
